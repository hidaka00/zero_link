# ZeroLink コネクタ要件書（v0.1 / MVP）

## 1. 目的
- ローカル環境で動作する AI処理間や処理間を、低遅延かつ汎用的に接続する「共有コネクタ基盤」を提供する。
- WebAPI/JSON 中心の連携よりリアルタイム性能を優先し、音声・映像・推論メタデータの連結を高速化する。
- アプリ間であったり処理ごとのプロセス分離したまま接続できる実行基盤を提供し、安定性・差し替え性・運用性を高める。

## 2. スコープ
- 対象: 単一ローカルPC内のプロセス間通信（IPC）。
- 対象: 同一PC上のアプリ/サービス/ノードなど、独立プロセス間の汎用的な接続と実行。
- 対象: `Node -> Connector -> Node` は代表ユースケースとしてサポートする。
- 非対象（MVP外）:
    - 複数マシン間通信（分散ネットワーク）。
    - 高度な UI ノードエディタ実装。
    - クラウド前提の認証/認可基盤。

## 3. 用語定義
- ノード: 入力を受けて処理し、出力を発行する独立プロセス。
- コネクタ: ノード間でデータ・イベント・制御を中継する共通機構。
- データ面: 共有メモリ上の重いデータ転送（音声/画像/tensor）。
- 制御面: 小さいメッセージ転送（参照ID、状態、コマンド等）。
- トピック: ルーティング単位（pub/sub または queue 名）。

## 4. 前提・設計方針
- ゼロコピーまたは準ゼロコピーを優先する。
- 大きい payload は共有メモリ参照で扱い、毎回コピーしない。
- 軽い制御情報は共通ヘッダ付きメッセージで扱う。
- ノードは障害分離のためプロセス分離を前提とする。
- 実装は初期段階から Rust を正とし、Connector 本体・SDK コアを Rust で開発する。
- 外部公開 API は C ABI 互換を維持し、多言語利用は C ABI を基準に提供する。

## 5. 機能要件
## 5.1 接続・転送
- F-01: Connector は topic ベースの `pub/sub` と `queue` の少なくとも一方を提供する（MVP は pub/sub 優先）。
- F-02: メッセージは共通ヘッダを持つ。最低項目は `type`, `timestamp`, `size`, `schema_id`, `trace_id`。
- F-03: payload は可変長を許容する。
- F-04: 重データは共有メモリ参照（buffer_id/offset/length）で受け渡し可能であること。
- F-05: イベント/制御メッセージは共有メモリを使わず送信可能であること。

## 5.2 データモデル
- F-06: MVP で以下3種の論理型を扱えること。
- `Frame`（音声フレーム、画像フレーム、tensor参照）
- `Event`（テキスト、bbox、推論結果）
- `Control`（start/stop/reload/config）
- F-07: スキーマ互換性判定用に `schema_id` を利用できること。
- F-16: MVP の `Event` / `Control` payload シリアライズは CBOR を標準とすること。

## 5.3 実行・信頼性
- F-08: producer/consumer は独立プロセスで接続できること。
- F-09: ノードの異常終了時、他ノードとコネクタ本体が継続動作可能であること。
- F-10: 再接続時に topic 購読を自動復旧できること（MVP では best effort）。
- F-11: バッファリーク検知のため、未解放参照を計測できること。

## 5.4 観測・デバッグ
- F-12: 各 topic の遅延（p50/p95）を取得できること。
- F-13: queue 深さ、ドロップ件数、スループットを取得できること。
- F-14: trace_id を使って段間追跡できること。
- F-15: 任意 topic の覗き見（サンプリング表示）機能を提供すること。

## 6. 非機能要件
## 6.1 性能
- N-01: Control/Event メッセージの片道遅延目標: 1ms 以下（同一マシン、平常時中央値）。
- N-02: Frame 連携はコピー回数 0〜1 回以内を目標とする。
- N-03: 連続ストリーム処理で安定動作し、バックプレッシャー制御を持つこと。

## 6.2 可用性・耐障害性
- N-04: 処理単体障害が全体停止に波及しないこと。
- N-05: Connector 再起動後に新規接続を受け付けられること。

## 6.3 拡張性
- N-06: 実装言語に依存しない ABI/API（C ABI など）を提供可能な設計にすること。
- N-07: 新しいデータ型追加時に既存ノードへの破壊的変更を最小化すること。
- N-10: Rust コアの内部変更時も C API シグネチャ互換を維持すること。
- N-11: Python/Rust ラッパは C ABI 互換層に追従し、コア改修の影響を吸収すること。

## 6.4 セキュリティ
- N-08: ローカル IPC のアクセス範囲を OS 権限で制御できること。
- N-09: 不正な schema_id/type/size を検出し破棄できること。

## 7. インタフェース要件（MVP）
- I-01: Node SDK の最小 API を提供する。
- `publish(topic, header, payload_or_buffer_ref)`
- `subscribe(topic, handler)`
- `alloc_buffer(size) / release_buffer(id)`
- `send_control(topic, command)`
- I-02: Connector 管理 API（CLI でも可）を提供する。
- topic 作成/削除、統計取得、購読一覧、ヘルス確認。

## 8. 制約
- C-01: 初期実装はローカルマシン限定。
- C-02: OS は Linux/Windows を対象とし、公開API/ABI/protocol の同等性を維持する。
- C-03: 共有メモリ容量の上限と eviction 方針を定義する。
- C-04: 0.x 系では API/ABI 変更があり得るが、破壊的変更時は互換方針と移行手順を必ず文書化する。
- C-05: topic 命名規約は `[a-z0-9_./-]` を許容し、予約プレフィックスは `_sys/` と `_dbg/` を使用する。

## 9. 受け入れ基準（MVP）
- A-01: `ASR Node -> Connector -> Parser Node` が連続入力で動作する。
- A-02: ノードを片側 kill しても、他ノードと Connector が継続稼働する。
- A-03: Frame は参照渡しで連携され、フルコピー転送が常態化しない。
- A-04: 遅延・スループット・ドロップ率のメトリクスを取得できる。
- A-05: trace_id により 2段以上の処理経路を追跡できる。

## 10. 今後の拡張（非MVP）
- 分散ノード連携（ネットワーク越し Connector）。
- GUI ベースのノード接続エディタ。
- レコード/リプレイ、時系列可視化との深い統合。
- スケジューラ最適化（優先度制御、GPU負荷連動）。

## 11. Rust開発方針（仕様更新）
- S-01: `connectord` は初版から Rust 実装を採用し、MVP 要件（F/N/A）を満たすことを優先する。
- S-02: C SDK の公開関数名/引数/戻り値の互換を維持し、Rust 実装を C ABI 経由で公開する。
- S-03: 開発ゲートは「機能要件テスト + 互換テスト + 性能目標（N-01〜N-03）」の通過とする。
- S-04: 多言語利用は C ABI を基準とし、Python/Rust ラッパの互換テストを必須化する。

## 12. Windows対応方針（仕様更新）
- W-01: 外部API/ABI（C ABI, protocol, packet format）は OS に依存せず共通化する。
- W-02: IPC 実装は OS 抽象層で分離し、Linux は UDS/`shm_open`、Windows は Named Pipe/File Mapping を採用する。
- W-03: 段階1で Windows ネイティブの実行/検証導線（PowerShell/CMD + native smoke/compat）を整備する。
- W-04: 段階2で Windows ネイティブ `connectord`/SDK を Linux と同等機能まで実装し、同一の互換テストを通す。
- W-05: 互換判定は Linux と Windows で同一の受け入れ基準（A-01〜A-05 + Rust開発DoD）を満たすこと。
